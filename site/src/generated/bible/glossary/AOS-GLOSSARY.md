## Purpose and Scope

This glossary defines specialized terms that carry architectural, hardware, camera, cellular, security, manufacturing, legal, product, evidence, or program meaning in Agent OS. Common development terms such as `commit`, `deploy`, and `firmware` are omitted unless the program assigns them a narrower contract.

Definitions are normative only to the extent required by the linked specification. Where an industry term varies by vendor or jurisdiction, the entry states the Agent OS interpretation and points to the governing paragraph.

<a id="how-to-use"></a>

## How to Use the Glossary

- Link to a term with `GLOSSARY.md#term-...`; do not duplicate a competing definition in another specification.
- Use the **Used in** links to find the exact normative paragraph or program section.
- Treat examples as illustrations, not as additional requirements unless the linked specification says otherwise.
- Add a term only with a definition, significance, example, and at least one exact specification reference.

## 0-9

<a id="term-three-a"></a>

### 3A (AE/AWB/AF)

**Definition.** The combined camera control loop for auto-exposure, auto-white-balance, and autofocus. Each controller consumes image or sensor statistics and produces settings for later frames, so latency, convergence, stability, and interaction between controllers matter.

**Why it matters.** A camera can produce valid RAW frames yet remain unusable when 3A oscillates, focuses slowly, or chooses wrong color and exposure.

**Example.** A scene becomes darker; AE lengthens exposure, AWB adapts to warmer light, and AF preserves the selected subject without hunting.

**Used in.** [AOS-HW-006#portable-pipeline](AOS-HW-006.md#portable-pipeline); [AOS-RES-003#camera-claims](AOS-RES-003.md#camera-claims)

<a id="letter-a"></a>

## A

<a id="term-a-b-update"></a>

### A/B update

**Definition.** An update layout with two independently bootable system slots or equivalent transactional images. A new image is written to the inactive side, verified, activated, health-checked, and rolled back when it cannot boot safely.

**Why it matters.** It reduces the probability that power loss or a bad image permanently removes the recovery path.

**Example.** Slot A remains known-good while version 12 is staged to slot B; the boot manager reverts to A after failed health checks.

**Used in.** [AOS-ARCH-013#image-layout](AOS-ARCH-013.md#image-layout); [AOS-ARCH-013#update-system](AOS-ARCH-013.md#update-system)

<a id="term-abi"></a>

### ABI (Application Binary Interface)

**Definition.** The binary-level contract between compiled code and a runtime or kernel: calling convention, register use, syscall numbers, data layout, alignment, symbol rules, executable format, and error representation.

**Why it matters.** An ABI can become a long-lived compatibility burden even when source APIs look clean, so Agent OS versions it deliberately.

**Example.** Changing a handle from 32 to 64 bits without an ABI level would break already compiled services.

**Used in.** [AOS-ARCH-005#api-levels](AOS-ARCH-005.md#api-levels); [AOS-GOV-001#api-levels](AOS-GOV-001.md#api-levels)

<a id="term-action-plan"></a>

### Action plan

**Definition.** A typed graph of proposed actions, dependencies, inputs, capabilities, budgets, checkpoints, confirmations, external effects, and compensation paths prepared before execution.

**Why it matters.** It makes agent behavior inspectable and separates planning from authority to act.

**Example.** A travel plan contains search, comparison, booking proposal, payment confirmation, calendar insertion, and rollback or support steps.

**Used in.** [AOS-PROD-007#planning](AOS-PROD-007.md#planning); [AOS-ARCH-010#action-plan](AOS-ARCH-010.md#action-plan)

<a id="term-action-provider"></a>

### Action provider

**Definition.** A component, remote service, device, or compatibility adapter that advertises typed actions and executes accepted requests under explicit capabilities.

**Why it matters.** Providers are the extensibility boundary; weak provider contracts would recreate application silos or ambient authority.

**Example.** A transit provider exposes route search and ticket purchase as separate actions with different effect classes.

**Used in.** [AOS-PROD-011#native-first](AOS-PROD-011.md#native-first); [AOS-PROD-003#provider-selection](AOS-PROD-003.md#provider-selection)

<a id="term-action-schema"></a>

### Action schema

**Definition.** A machine-readable contract for a user-meaningful operation, including typed inputs and outputs, required capabilities, provider, effects, idempotency, reversibility, confirmation, cost, data destinations, and receipt behavior.

**Why it matters.** It lets the shell, agents, providers, policy engine, and audit system reason about the same operation without hidden app-specific behavior.

**Example.** A `message.send` action identifies recipient, account, body, external effect, confirmation policy, and non-reversible delivery receipt.

**Used in.** [AOS-PROD-003#action-schema](AOS-PROD-003.md#action-schema); [AOS-ARCH-010#action-plan](AOS-ARCH-010.md#action-plan)

<a id="term-address-space-identifier"></a>

### Address-space identifier (ASID)

**Definition.** A processor-supported tag associated with an address space so translations from multiple processes can coexist in the translation lookaside buffer without flushing everything at each context switch.

**Why it matters.** Correct allocation and reuse are necessary for both performance and isolation; stale ASID reuse can expose old mappings.

**Example.** Before reassigning an ASID to another process, the kernel invalidates translations belonging to its previous owner.

**Used in.** [AOS-ARCH-003#address-spaces](AOS-ARCH-003.md#address-spaces); [AOS-ARCH-003#architecture-portability](AOS-ARCH-003.md#architecture-portability)

<a id="term-agent-shadow-mode"></a>

### Agent shadow mode

**Definition.** An evaluation mode in which an agent observes permitted context and records proposed plans without producing external effects.

**Why it matters.** It permits measurement of usefulness and dangerous false positives before execution privileges are granted.

**Example.** The agent predicts which messages to send and appointments to create, but only compares its proposals with the user’s later actions.

**Used in.** [AOS-ARCH-010#shadow-mode](AOS-ARCH-010.md#shadow-mode); [AOS-VAL-005#agents](AOS-VAL-005.md#agents)

<a id="term-ambient-authority"></a>

### Ambient authority

**Definition.** Authority available implicitly from process identity, global namespaces, inherited environment, or unrestricted system calls rather than from an explicit object-specific capability.

**Why it matters.** Ambient authority makes dependency and security review difficult and creates confused-deputy paths.

**Example.** A library that can open any path by name has ambient filesystem authority; a library given one document handle does not.

**Used in.** [AOS-ARCH-012#least-authority](AOS-ARCH-012.md#least-authority); [AOS-ARCH-004#confused-deputy](AOS-ARCH-004.md#confused-deputy)

<a id="term-anti-circumvention"></a>

### Anti-circumvention

**Definition.** Legal rules governing bypass of technological access controls, distinct from ordinary copyright infringement. Exceptions and exemptions are jurisdiction-, purpose-, device-, method-, and time-specific and do not automatically override contracts, patents, trade secrets, or redistribution limits.

**Why it matters.** Pixel and undocumented-protocol work can be technically possible yet legally impermissible or publishable only under narrower conditions.

**Example.** A smartphone interoperability exemption may permit a particular unlock activity but not redistribution of proprietary firmware or leaked documentation.

**Used in.** [AOS-LEGAL-001#device-and-protocol-work](AOS-LEGAL-001.md#device-and-protocol-work); [AOS-LEGAL-002#clean-room-principle](AOS-LEGAL-002.md#clean-room-principle)

<a id="term-anti-rollback"></a>

### Anti-rollback

**Definition.** A security mechanism that prevents installation or boot of software below an approved security version, even when an older image is correctly signed.

**Why it matters.** A signature alone cannot stop an attacker from restoring a known-vulnerable signed build.

**Example.** The boot verifier accepts release 15 but rejects signed release 12 after the device security version has advanced.

**Used in.** [AOS-ARCH-013#anti-rollback](AOS-ARCH-013.md#anti-rollback); [AOS-ARCH-013#migration-and-rollback](AOS-ARCH-013.md#migration-and-rollback)

<a id="term-apdu"></a>

### APDU (Application Protocol Data Unit)

**Definition.** A command/response unit used to communicate with smart-card applications such as a SIM, eUICC, or secure element. Raw APDU access can expose highly privileged functions and application secrets.

**Why it matters.** The cellular service should broker narrow SIM operations instead of giving every client an unrestricted card channel.

**Example.** The trusted SIM service verifies a PIN command while an ordinary messaging provider receives only subscriber-state information.

**Used in.** [AOS-HW-007#portable-cellular-contract](AOS-HW-007.md#portable-cellular-contract); [AOS-ARCH-012#keys-and-secrets](AOS-ARCH-012.md#keys-and-secrets)

<a id="term-assurance-case"></a>

### Assurance case

**Definition.** A structured argument linking security or safety claims to assumptions, architecture controls, implementation, tests, analyses, independent reviews, and residual risks for an exact version and target.

**Why it matters.** It prevents broad words such as “secure” from outliving the evidence that originally supported them.

**Example.** The kernel isolation claim cites the memory model, fault tests, fuzzing results, reviewed code revision, and unsupported DMA case.

**Used in.** [AOS-ARCH-012#assurance-roadmap](AOS-ARCH-012.md#assurance-roadmap); [AOS-RES-003#claim-states](AOS-RES-003.md#claim-states)

<a id="term-attestation"></a>

### Attestation

**Definition.** A cryptographically authenticated statement about device identity, boot measurements, software state, configuration, or security mode, usually produced by a protected key and checked by a verifier.

**Why it matters.** Attestation can support trust decisions but can also create tracking and overclaim risks if its scope and privacy are vague.

**Example.** A verifier learns that a device booted an approved development image, but not the user’s global identity or unrelated application history.

**Used in.** [AOS-ARCH-013#verified-boot](AOS-ARCH-013.md#verified-boot); [AOS-ARCH-012#keys-and-secrets](AOS-ARCH-012.md#keys-and-secrets)

<a id="term-authority-amplification"></a>

### Authority amplification

**Definition.** A flaw in which a component can transform limited authority into broader authority, often through a privileged intermediary, confused deputy, weak object derivation, or unchecked provider routing.

**Why it matters.** Capability systems are useful only when delegation is monotonic and intermediaries cannot silently act with their own stronger privilege.

**Example.** A thumbnail service must not use its broad media-store access to return arbitrary files requested through an untrusted path string.

**Used in.** [AOS-ARCH-004#derivation-revocation](AOS-ARCH-004.md#derivation-revocation); [AOS-ARCH-004#confused-deputy](AOS-ARCH-004.md#confused-deputy)

<a id="term-autonomy-budget"></a>

### Autonomy budget

**Definition.** A bounded allowance for agent time, actions, money, network destinations, data fields, energy, or other resources within a defined context.

**Why it matters.** Autonomy must be constrained by more than a broad permission grant.

**Example.** A purchasing agent may compare offers for ten minutes and spend up to a stated amount, but cannot add new recipients or payment instruments.

**Used in.** [AOS-ARCH-010#agent-budgets](AOS-ARCH-010.md#agent-budgets); [AOS-PROD-005#interaction-principles](AOS-PROD-005.md#interaction-principles)

<a id="letter-b"></a>

## B

<a id="term-backend-conformance"></a>

### Backend conformance

**Definition.** Evidence that a target-specific implementation satisfies the same native service semantics and negative behavior as other backends.

**Why it matters.** Portability is meaningful only when backends are interchangeable at a tested contract boundary.

**Example.** QEMU and an AArch64 board pass identical display-buffer ownership, cancellation, hotplug, and permission tests.

**Used in.** [AOS-ARCH-016#conformance](AOS-ARCH-016.md#conformance); [AOS-VAL-001#portable](AOS-VAL-001.md#portable)

<a id="term-backpressure"></a>

### Backpressure

**Definition.** A flow-control mechanism that causes producers to slow, block, drop according to policy, or receive an explicit error when consumers or buffers cannot keep up.

**Why it matters.** Without backpressure, camera, network, logging, and IPC paths can exhaust memory or hide data loss.

**Example.** A camera stream reports a dropped-frame reason or blocks capture rather than allocating unbounded buffers when an encoder stalls.

**Used in.** [AOS-ARCH-004#streaming](AOS-ARCH-004.md#streaming); [AOS-ARCH-008#buffer-model](AOS-ARCH-008.md#buffer-model)

<a id="term-baseband"></a>

### Baseband

**Definition.** The cellular modem subsystem that implements radio protocols and often runs its own firmware and processors, isolated to varying degrees from the application processor.

**Why it matters.** It is a complex external trust domain; packet data support does not expose or solve its voice, IMS, emergency, or certification functions.

**Example.** Agent OS controls a modem through a documented transport while treating modem firmware as an untrusted peripheral with reset and update policy.

**Used in.** [AOS-HW-007#program-boundary](AOS-HW-007.md#program-boundary); [AOS-ARCH-012#trust-boundaries](AOS-ARCH-012.md#trust-boundaries)

<a id="term-binary-blob"></a>

### Binary blob

**Definition.** A precompiled artifact whose complete source or build process is unavailable. It may be redistributable, device-bound, NDA-restricted, or legally non-redistributable; those categories are not interchangeable.

**Why it matters.** A blob can be an acceptable bounded dependency only when rights, provenance, update, security, lifecycle, and replacement are explicit.

**Example.** A signed camera firmware binary is recorded by hash and license, loaded only by its backend, and excluded from public packages when redistribution is not permitted.

**Used in.** [AOS-ARCH-006#firmware-policy](AOS-ARCH-006.md#firmware-policy); [AOS-LEGAL-003#provenance-record](AOS-LEGAL-003.md#provenance-record)

<a id="term-black-level"></a>

### Black level

**Definition.** The sensor output value representing no light after accounting for electronic offset. It can vary by sensor mode, channel, gain, exposure, temperature, and row/column behavior.

**Why it matters.** Incorrect black-level correction causes color casts, crushed shadows, banding, and invalid noise measurements.

**Example.** Calibration records per-channel black levels and subtracts them before demosaic and color processing.

**Used in.** [AOS-HW-006#calibration-lab](AOS-HW-006.md#calibration-lab); [AOS-HW-006#metrics](AOS-HW-006.md#metrics)

<a id="term-black-box-observation"></a>

### Black-box observation

**Definition.** The study of a system through controlled inputs and externally observable outputs without using protected internal source or implementation details.

**Why it matters.** It is a common building block for lawful interoperability work and produces behavioral tests rather than copied expression.

**Example.** The observation team changes a modem state, records approved response fields and timing, then writes a functional state specification for a separate implementation team.

**Used in.** [AOS-LEGAL-002#workflow](AOS-LEGAL-002.md#workflow); [AOS-LEGAL-002#roles](AOS-LEGAL-002.md#roles)

<a id="term-board-dossier"></a>

### Board dossier

**Definition.** A versioned record for a specific board or device revision covering boot, recovery, debug, schematics, documentation, firmware, interfaces, supply, legal terms, risks, experiments, and support level.

**Why it matters.** Board names alone hide revision and access differences that can invalidate a port plan.

**Example.** The dossier records UART voltage, recovery straps, exact PMIC, image procedure, public TRM revision, and unavailable ISP documents.

**Used in.** [AOS-HW-011#dossier](AOS-HW-011.md#dossier); [AOS-HW-004#documentation-audit](AOS-HW-004.md#documentation-audit)

<a id="term-board-package"></a>

### Board package

**Definition.** The target-specific description and backend collection that translates a board’s boot data, buses, MMIO, interrupts, clocks, resets, power rails, pins, DMA/IOMMU, devices, firmware, and quirks into portable Agent OS resource and service contracts.

**Why it matters.** It confines target variation and allows product and system services to remain unchanged across unrelated boards.

### IOVA (I/O Virtual Address)

**Definition.** An address visible to a DMA-capable device after translation by an IOMMU. It is distinct from a process virtual address and from the underlying physical address.

**Why it matters.** Correct lifetime, ownership, cache, alignment, and unmap semantics prevent devices from retaining access to recycled memory.

**Example.** The GPU service receives an IOVA scoped to its isolation domain; after the job fence signals, the mapping is released before the memory object is delegated elsewhere.

<a id="term-ipc"></a>

### IPC (Inter-Process Communication)

**Definition.** The message, call, queue, shared-memory, signaling, and capability-transfer mechanisms used by isolated processes or services to cooperate.

**Why it matters.** IPC semantics define authority transfer, backpressure, cancellation, priority behavior, failure containment, and much of a microkernel system’s performance envelope.

**Example.** A camera client sends a bounded request over a channel and receives a capability to a shared buffer collection plus a completion event.

**Used in.** [AOS-ARCH-004#ipc-model](AOS-ARCH-004.md#ipc-model); [AOS-ARCH-004#message-invariants](AOS-ARCH-004.md#message-invariants)

<a id="term-irq"></a>

### IRQ (Interrupt Request)

**Definition.** A hardware event delivered through an interrupt controller to notify software of device or timer conditions. IRQ handling requires explicit routing, acknowledgement, masking, affinity, storm control, and handoff to user-space drivers.

**Why it matters.** Interrupt mistakes can deadlock a device, starve the system, lose events, or allow one driver to interfere with another.

**Example.** The kernel acknowledges the controller, signals the authorized driver domain, and masks a repeated level interrupt until the driver clears the device condition.

**Example.** The test generates grant/revoke/transfer sequences and asserts that no descendant ever obtains rights absent from its ancestor.

<a id="term-suspend-to-idle"></a>

### Suspend-to-idle

**Definition.** A low-power state in which most user work stops and devices enter reduced-power modes while memory remains powered and selected wake sources can resume the system quickly.

**Why it matters.** Phone battery life depends on coordinated service quiescence, modem and sensor state, wake filtering, timers, and measurement—not only CPU sleep.

**Example.** Before suspend, services publish blockers, the power manager arms approved wake sources, and the evidence records residency and unexpected wakes.

**Used in.** [AOS-ARCH-014#system-suspend](AOS-ARCH-014.md#system-suspend); [AOS-ARCH-014#background-work](AOS-ARCH-014.md#background-work)

<a id="letter-t"></a>

## T

<a id="term-taint-boundary"></a>

### Taint boundary

**Definition.** A repository, role, interface, or process boundary that prevents restricted or disallowed source material from influencing independent implementation.

**Why it matters.** Clean-room and contractual obligations require enforceable separation, not merely contributor intent.

**Example.** A specification team may see behavioral observations while implementation contributors receive only the approved independent interface specification.

**Used in.** [AOS-ARCH-018#taint](AOS-ARCH-018.md#taint); [AOS-LEGAL-002#roles](AOS-LEGAL-002.md#roles)

<a id="term-target-dossier"></a>

### Target dossier

**Definition.** A decision-grade package for one hardware target, broader than a board dossier when it includes product quality, partner, supply, legal, cost, certification, and stop criteria.

**Why it matters.** Purchases and ports should follow comparable evidence rather than enthusiasm for a chipset or brand.

**Example.** The Pixel 9 dossier combines recovery, boot, display, GPU, power, camera, modem, firmware, rights, cost, and replacement evidence.

**Used in.** [AOS-HW-014#inventory](AOS-HW-014.md#inventory); [AOS-HW-011#dossier](AOS-HW-011.md#dossier)

<a id="term-tcb"></a>

### TCB (Trusted Computing Base)

**Definition.** The set of hardware, firmware, software, keys, people, and procedures whose compromise can violate a stated security property.

**Why it matters.** Calling only the microkernel the TCB hides boot firmware, DMA, update signing, secure elements, critical services, and manufacturing operations that may be equally trusted.

**Example.** The data-confidentiality TCB includes the kernel, memory controller/IOMMU, storage encryption service, boot chain, key service, and recovery path.

**Used in.** [AOS-ARCH-012#trust-boundaries](AOS-ARCH-012.md#trust-boundaries); [AOS-ARCH-012#assurance-roadmap](AOS-ARCH-012.md#assurance-roadmap)

<a id="term-tee"></a>

### TEE (Trusted Execution Environment)

**Definition.** An isolated execution environment supported by hardware and privileged firmware, commonly used for keys, attestation, biometrics, digital rights, or sensitive services. Its security and ownership depend on the platform implementation.

**Why it matters.** A TEE may reduce exposure but can introduce opaque firmware, vendor signing, limited debugging, and lifecycle dependencies.

**Example.** Agent OS sends a narrowly defined key-use request to a TEE service and treats the TEE as an explicit external trust domain, not as ordinary RAM.

**Used in.** [AOS-ARCH-012#trust-boundaries](AOS-ARCH-012.md#trust-boundaries); [AOS-ARCH-012#keys-and-secrets](AOS-ARCH-012.md#keys-and-secrets)

<a id="term-thermal-envelope"></a>

### Thermal envelope

**Definition.** The combinations of ambient conditions, device temperature, power, workload duration, enclosure, charging, radios, and cooling within which required performance and safety behavior can be sustained.

**Why it matters.** Peak benchmark performance says little about camera recording, navigation, modem use, charging, or agent workloads over time.

**Example.** The quality gate measures 4K recording plus cellular upload at defined ambient temperature until steady state, recording frame drops and skin limits.

**Used in.** [AOS-ARCH-014#thermal-management](AOS-ARCH-014.md#thermal-management); [AOS-HW-010#weighted-categories](AOS-HW-010.md#weighted-categories)

<a id="term-tlb"></a>

### TLB (Translation Lookaside Buffer)

**Definition.** A processor cache of recent virtual-to-physical address translations and permissions.

**Why it matters.** Page-table changes, address-space identifiers, revocation, DMA, and multicore shootdown must account for cached translations or stale access can persist.

**Example.** After removing an executable mapping, the kernel invalidates affected TLB entries before reusing the physical page for another process.

**Used in.** [AOS-ARCH-003#address-spaces](AOS-ARCH-003.md#address-spaces); [AOS-ARCH-003#architecture-portability](AOS-ARCH-003.md#architecture-portability)

<a id="term-toctou"></a>

### TOCTOU (Time of Check to Time of Use)

**Definition.** A race in which a security or validity condition is checked and then changes before the protected operation uses it.

**Why it matters.** Pathnames, grants, buffers, manifests, device state, and user confirmations can be swapped or revoked between planning and execution.

**Example.** The action broker binds confirmation to an immutable action-plan hash and the provider revalidates the live capability at commit time.

**Used in.** [AOS-ARCH-004#message-invariants](AOS-ARCH-004.md#message-invariants); [AOS-ARCH-010#confirmation-policy](AOS-ARCH-010.md#confirmation-policy)

<a id="term-tone-mapping"></a>

### Tone mapping

**Definition.** The mapping of captured or scene-referred luminance and color into the limited range of an output image or display, often with global and local contrast decisions.

**Why it matters.** Tone mapping strongly shapes perceived mobile-camera quality and can create halos, flattened faces, clipped highlights, or inconsistent video when unstable.

**Example.** The HDR pipeline applies a scene-adaptive curve with temporal smoothing and evaluates skin, highlight roll-off, and local contrast on the scene corpus.

**Used in.** [AOS-HW-006#portable-pipeline](AOS-HW-006.md#portable-pipeline); [AOS-HW-006#metrics](AOS-HW-006.md#metrics)

<a id="term-trace-harness"></a>

### Trace harness

**Definition.** Infrastructure that records, normalizes, replays, compares, and validates service-level event traces against native contracts.

**Why it matters.** It turns external behavior into repeatable tests without adopting the external implementation ABI.

**Example.** A recorded camera lifecycle is replayed in QEMU to test cancellation, metadata ordering, and device-loss behavior.

**Used in.** [AOS-ARCH-020#mocking](AOS-ARCH-020.md#mocking); [AOS-VAL-002#automation](AOS-VAL-002.md#automation)

<a id="term-trace-replay"></a>

### Trace replay

**Definition.** The deterministic feeding of recorded, sanitized hardware or service events into a native implementation to reproduce timing, state transitions, errors, and regressions without executing the original legacy stack.

**Why it matters.** It converts Pixel, modem, camera, power, and input observations into portable tests while preserving the Android/Linux containment boundary.

**Example.** A recorded stock-modem sequence is replayed against the Agent OS state machine and compared with the expected registration and failure transitions.

**Used in.** [AOS-ARCH-011#trace-replay](AOS-ARCH-011.md#trace-replay); [AOS-ARCH-015#test-pyramid](AOS-ARCH-015.md#test-pyramid)

<a id="term-transactional-update"></a>

### Transactional update

**Definition.** An update design in which image installation, verification, activation, migration, health confirmation, rollback, and cleanup have explicit atomic or recoverable boundaries.

**Why it matters.** Power loss or crash during update must not leave an unverifiable mixture of old and new code or incompatible data.

**Example.** The inactive slot is written and verified, activation is committed in protected metadata, and the previous slot remains available until health checks pass.

**Used in.** [AOS-ARCH-013#update-system](AOS-ARCH-013.md#update-system); [AOS-ARCH-013#migration-and-rollback](AOS-ARCH-013.md#migration-and-rollback)

<a id="term-trust-ladder"></a>

### Trust ladder

**Definition.** A graduated authorization model that increases an agent’s autonomy only as scope, reversibility, sensitivity, evidence, user confirmation, and policy permit.

**Why it matters.** Treating every agent as either read-only or fully autonomous produces poor usability or unacceptable risk.

**Example.** An agent may observe calendar state, propose a trip, reserve a refundable option under a budget, but must obtain confirmation before a non-refundable purchase.

**Used in.** [AOS-ARCH-010#trust-ladder](AOS-ARCH-010.md#trust-ladder); [AOS-ARCH-010#confirmation-policy](AOS-ARCH-010.md#confirmation-policy)

<a id="term-trust-tier"></a>

### Trust tier

**Definition.** A context-specific level controlling whether an agent may observe, propose, execute reversible actions, request sensitive confirmation, or act autonomously within budgets.

**Why it matters.** Trust must be narrower than a global “agent enabled” switch and can vary by provider, data domain, and effect.

**Example.** An agent may auto-file local notes but only propose a bank transfer and require explicit biometric confirmation for execution.

**Used in.** [AOS-ARCH-010#trust-ladder](AOS-ARCH-010.md#trust-ladder); [AOS-VAL-005#agents](AOS-VAL-005.md#agents)

<a id="term-tuning-profile"></a>

### Tuning profile

**Definition.** Versioned parameters, curves, matrices, models, thresholds, calibration references, and control policies that adapt a camera or other subsystem to a specific sensor, lens, board, mode, temperature, and quality target.

**Why it matters.** Tuning is executable product knowledge and must have provenance, licensing, validation, versioning, and rollback like code.

**Example.** The camera package selects a signed profile by module serial range and sensor mode, then records its hash in image metadata and evidence.

**Used in.** [AOS-HW-006#calibration-lab](AOS-HW-006.md#calibration-lab); [AOS-ARCH-013#migration-and-rollback](AOS-ARCH-013.md#migration-and-rollback)

<a id="letter-u"></a>

## U

<a id="term-uefi"></a>

### UEFI (Unified Extensible Firmware Interface)

**Definition.** A standardized firmware interface and boot environment used on many systems to discover devices, load images, expose boot services, and hand control to an operating system.

**Why it matters.** UEFI can accelerate QEMU and development-board bootstrapping, but Agent OS must still define its post-handoff memory map, trust, device ownership, and architecture-native boot path.

**Example.** The x86_64 port receives the image and memory map from UEFI, exits boot services, validates ownership, and initializes Agent OS page tables before starting the kernel.

**Used in.** [AOS-ARCH-013#boot-chain](AOS-ARCH-013.md#boot-chain); [AOS-ARCH-003#architecture-portability](AOS-ARCH-003.md#architecture-portability)

<a id="term-user-space-driver"></a>

### User-space driver

**Definition.** A hardware driver that executes outside the kernel in an isolated process or domain and accesses only delegated MMIO, interrupts, DMA, clocks, and related resources through kernel-mediated capabilities.

**Why it matters.** It reduces kernel TCB and enables restart and fault attribution, but requires secure resource brokering, DMA isolation, lifecycle protocols, and performance measurement.

**Example.** The USB host controller driver crashes; its domain is restarted and re-enumerates devices without granting it unrelated storage registers.

**Used in.** [AOS-ARCH-006#driver-model](AOS-ARCH-006.md#driver-model); [AOS-ARCH-006#driver-lifecycle](AOS-ARCH-006.md#driver-lifecycle)

<a id="term-uvc"></a>

### UVC (USB Video Class)

**Definition.** A standardized USB device class for cameras and video sources, defining descriptors, streaming interfaces, formats, controls, and negotiation behavior.

**Why it matters.** UVC provides a practical documented camera bench and early physical capture path without committing to a SoC MIPI/ISP stack.

**Example.** The board enumerates a UVC camera, negotiates a YUYV stream, and presents frames through the same portable camera service used by native sensors.

**Used in.** [AOS-HW-006#parallel-camera-paths](AOS-HW-006.md#parallel-camera-paths); [AOS-ARCH-006#portable-development](AOS-ARCH-006.md#portable-development)

<a id="term-uwb"></a>

### UWB (Ultra-Wideband)

**Definition.** A short-range radio technology using very wide bandwidth for ranging, positioning, discovery, or data functions, with region-specific spectrum and product requirements.

**Why it matters.** A quality phone may include UWB, but native support requires radio control, secure ranging, calibration, coexistence, privacy, and certification—not just device enumeration.

**Example.** The service exposes a consented ranging session with bounded peers while the hardware backend owns calibration and regional channel policy.

**Used in.** [AOS-HW-007#wireless-and-gnss](AOS-HW-007.md#wireless-and-gnss); [AOS-LEGAL-004#scope](AOS-LEGAL-004.md#scope)

<a id="letter-v"></a>

## V

<a id="term-vector-clock"></a>

### Vector clock

**Definition.** A logical timestamp containing one counter per participant or replica, allowing a system to distinguish causal ordering from concurrency among events.

**Why it matters.** It provides richer conflict information than a scalar clock but increases metadata and participant-management complexity.

**Example.** Two offline devices produce incomparable vector clocks, so the merge policy preserves both edits and records a user-resolvable conflict.

**Used in.** [AOS-ARCH-009#local-first-sync](AOS-ARCH-009.md#local-first-sync); [AOS-ARCH-009#crdt-policy](AOS-ARCH-009.md#crdt-policy)

<a id="term-verified-boot"></a>

### Verified boot

**Definition.** A boot chain that cryptographically authenticates executable and policy artifacts before use and binds acceptance to a root-of-trust and rollback policy.

**Why it matters.** It protects integrity but does not by itself guarantee confidentiality, absence of vulnerabilities, correct recovery, or user ownership of the signing key.

**Example.** The boot manager verifies the signed manifest, each image hash, configuration digest, and rollback epoch before launching the system slot.

**Used in.** [AOS-ARCH-013#verified-boot](AOS-ARCH-013.md#verified-boot); [AOS-ARCH-013#anti-rollback](AOS-ARCH-013.md#anti-rollback)

<a id="term-vsync"></a>

### Vsync (Vertical Synchronization)

**Definition.** A display timing event or prediction associated with refresh boundaries, used to schedule composition, buffer presentation, input sampling, and frame pacing.

**Why it matters.** Ignoring vsync causes tearing, uneven latency, missed frames, and excess power; blindly waiting can also add latency.

**Example.** The compositor predicts the next presentation deadline and starts rendering early enough to submit the finished buffer before scan-out.

**Used in.** [AOS-ARCH-008#performance](AOS-ARCH-008.md#performance); [AOS-ARCH-008#display-and-color](AOS-ARCH-008.md#display-and-color)

<a id="letter-w"></a>

## W

<a id="term-wake-source"></a>

### Wake source

**Definition.** A hardware or software event authorized to resume a suspended system, such as a power button, timer, modem indication, charger, alarm, or selected sensor condition.

**Why it matters.** Unbounded wake sources destroy battery life; overly strict policy misses calls, alarms, safety events, or user input.

**Example.** The power manager arms the modem and alarm timer but suppresses a noisy sensor until its driver clears the fault and records the wake reason.

**Used in.** [AOS-ARCH-014#system-suspend](AOS-ARCH-014.md#system-suspend); [AOS-ARCH-014#background-work](AOS-ARCH-014.md#background-work)

<a id="term-wire-format"></a>

### Wire format

**Definition.** The exact byte-level representation, alignment, framing, endianness, tags, handles, limits, and evolution rules used to exchange an IDL message or stored record.

**Why it matters.** Language-level types are not interoperability; a stable and validated wire format is required across processes, versions, architectures, and generated bindings.

**Example.** An older service skips an unknown optional field but rejects a message whose declared handle count exceeds the bounded envelope.

**Used in.** [AOS-ARCH-005#type-system](AOS-ARCH-005.md#type-system); [AOS-ARCH-005#versioning](AOS-ARCH-005.md#versioning)

<a id="term-working-set"></a>

### Working set

**Definition.** The subset of code and data actively needed over a time interval, often used to reason about memory pressure, caching, paging, launch latency, and background retention.

**Why it matters.** Mobile systems have tight memory and power budgets; retaining every agent, surface, or cache harms foreground quality and may increase storage churn.

**Example.** The process manager trims a background provider’s reclaimable cache while retaining its semantic checkpoint and restart metadata.

**Used in.** [AOS-ARCH-003#resource-accounting](AOS-ARCH-003.md#resource-accounting); [AOS-ARCH-014#background-work](AOS-ARCH-014.md#background-work)

<a id="letter-z"></a>

## Z

<a id="term-zero-copy"></a>

### Zero-copy

**Definition.** A data path that avoids one or more CPU-mediated payload copies by sharing, transferring, mapping, or directly producing into controlled buffers while preserving ownership and synchronization.

**Why it matters.** Camera, display, audio, networking, and storage throughput benefit, but zero-copy can expand authority, pin memory, expose stale data, and complicate lifecycle.

**Example.** The camera produces into a buffer collection accepted by the compositor; fences and capability rights prevent reuse until every consumer releases the frame.

**Used in.** [AOS-ARCH-004#streaming](AOS-ARCH-004.md#streaming); [AOS-ARCH-008#buffer-model](AOS-ARCH-008.md#buffer-model)
